# PoC 運用ルール

目的: Claude Codeのトークン消費の重い箇所を特定し、他エージェントへの置き換え候補を洗い出す。

---

## 1. タスクログの記録

タスク完了直後（/clear前）に `docs/poc-log.md` へ1行追記する。

| 記録項目 | 選択肢 |
|---|---|
| フロー | fast / flow1 |
| overview操作 | skip / read / update |
| Codex失敗回数 | 0 / 1 / 2以上 |
| Geminiレビュー | skip / run |
| 体感の重さ | 軽 / 中 / 重 |
| セッション内の何件目か | 数字 |

「体感の重さ」の判断基準:
- **軽**: Claude Codeが1〜2ターンで設計書を出力し、すぐ委任できた
- **中**: 3〜4ターン、または途中でファイルを読み直した
- **重**: 5ターン以上、またはCodex失敗による設計見直しが発生した

---

## 2. /clear のタイミング

**タスク完了後は必ず /clear する（既定ルール）。**

加えて、以下のいずれかが発生したら**タスク完了を待たず即座に /clear して仕切り直す**:

- Claude Codeが同じファイルを2回以上読み直した
- 設計書の内容について Claude Code がすでに書いたことを再確認し始めた
- 1タスクで5ターンを超えた（タスクが大きすぎる可能性）

仕切り直し後は、タスクをより小さく分割して再開する。

---

## 3. 調整のタイミング

| フェーズ | タスク数 | やること |
|---|---|---|
| Phase 1 | 1〜5件 | **観測のみ。設定変更禁止** |
| Phase 2 | 6〜10件 | ログから最も重い1箇所を特定し、1つだけ調整する |
| Phase 3 | 11件〜 | 調整効果を確認し、安定したら次の重箇所へ |

**1回の調整で変える設定は1つだけ。**複数同時変更は効果の切り分けができなくなる。

---

## Phase 2 の調整候補（観測結果に応じて選ぶ）

| 観測パターン | 調整内容 |
|---|---|
| overviewが毎タスク更新されている | キャッシュ更新トリガー条件を厳しくする |
| Codex失敗が多い（1タスク1回以上） | 大枠設計書の制約セクションをより具体的に書く |
| ファストパス対象が多い | ファストパス判定基準を広げる |
| レビューが頻繁に発動している | スキップ基準を定量化する |
| 体感「重」が多い | タスク分割基準をさらに厳しくする |
