フロー1（フルパイプライン）を適用する。

## タスク分割の判断

以下のいずれかに該当する場合はタスクを分割してから着手する。

- 変更ファイル 4 以上
- FE/BE 横断
- 新規エンドポイント + 新規コンポーネントの同時発生

## フロー

```
① 大枠設計（Claude Code）
   - docs/overview-*.md キャッシュ確認（探索禁止）
   - 設計書作成: docs/design-{yyyymmdd}-{index}.md（40行以内）
   - セクション: 目的 / 変更対象ファイル / インターフェース定義 / 実装方針 / 制約
   └→ claude_to_codex/ に指示ファイル出力

② 詳細設計・テスト設計（codex-design）
   - 大枠設計書末尾に追記（採用しなかった選択肢と理由 / 依存関係・注意事項）
   - テスト設計書: docs/test-design-{yyyymmdd}-{index}.md（200行以内）
   └→ codex_to_codex/ に引き継ぎ出力

③ 実装・テスト（codex-test-impl）
   - 修正ループ上限: 3回超過 → 中断・ERROR報告
   - テストコード変更禁止（実装コードのみ修正）
   └→ codex_to_claude/ に完了報告出力

④ コードレビュー（選択的）
   - 必須: 認証・セキュリティ・DB操作・APIスキーマ変更
   - スキップ可: UI文言・スタイル・1〜2ファイル軽微ロジック変更
   └→ claude_to_gemini/ に指示 → gemini_to_claude/ 受取
```

## レビュー結果の対処

| 指摘内容 | 対処 |
|---|---|
| typo・命名・フォーマットのみ | Claude Code が直接修正（再レビュー不要） |
| ロジック修正を伴う指摘 | codex-test-impl に修正委任 → 再レビュー実施 |

## コードベース参照制限

- 原則禁止（最大 2ファイル/ターン）
- overview キャッシュ活用が前提

## 完了

`/clear` を実行してから次のタスクを受け付ける。
